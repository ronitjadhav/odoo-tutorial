## Why this matters

You're about to spend weeks learning a framework. Before touching code, you need to
know what the thing actually *is* — because Odoo's architecture only makes sense once
you understand its business model: one company selling a suite of business apps that
thousands of other companies customize for their clients. Nearly every design choice
you'll meet later (modules, inheritance everywhere, views stored as data) exists to
make that customization industry possible.

## Concepts

### ERP, in one paragraph

An ERP (Enterprise Resource Planning) system is the software a company runs its
operations on: sales, invoicing, inventory, accounting, HR — all sharing **one
database**, so a confirmed sale automatically reserves stock, creates an invoice, and
shows up in the accounting reports. The alternative is a zoo of disconnected tools glued
together with spreadsheets and re-typing. The single shared database is the entire value
proposition.

### Odoo's take on it

Odoo is a suite of open-source business apps — CRM, Sales, Inventory, Accounting,
Website, Manufacturing, HR, and dozens more — built on one shared framework and one
PostgreSQL database. Two things make it different from classical ERPs:

1. **It's modular to the bone.** Every app is a *module*, and so is every
   customization. Installing "Sales" and installing a 200-line extension you wrote
   yourself use exactly the same mechanism.
2. **The framework is a product too.** You get a full web framework (ORM, views,
   reports, access control, web client) that you'll use to build things that look and
   feel native. That framework is what this tutorial teaches.

### Apps vs modules

There is no technical difference. An **app** is just a module with
`'application': True` in its manifest, which makes it show up on the Apps home screen
with an icon. Modules that extend other modules (most of what you'll write) don't set
that flag. When Odoo people say *addon*, they also mean module — you'll see
`--addons-path` as the flag that tells the server where to find them.

### Community vs Enterprise

Odoo comes in two editions:

- **Community** — open source (LGPL-3), the complete framework and core apps. Its
  source lives publicly at [github.com/odoo/odoo](https://github.com/odoo/odoo). This
  is what the OCA targets and what this tutorial uses.
- **Enterprise** — a paid subscription that adds apps and features on top of Community
  (Studio, full Accounting localizations, Field Service, upgrade service, mobile
  apps...). Its source is visible to partners and customers but is **not** freely
  redistributable.

<Callout title="Enterprise is a superset">
  Enterprise *contains* Community — it's the same framework with more modules on top.
  Everything you learn here transfers directly to Enterprise projects; we'll flag
  Enterprise-only features in boxes like this when they matter.
</Callout>

### The version rhythm — there is no LTS

Odoo ships **one major version every year, around October**, announced at the Odoo
Experience conference in Brussels. Only the **three most recent** majors receive
support and security fixes. As this is written (mid-2026), that window is **17, 18
and 19**, with Odoo 20 expected in October 2026 — at which point 17 falls out.

This has two consequences you'll feel throughout your career:

- Clients must **migrate** roughly every three years, whether they like it or not.
  Migration work is a big part of integrator life — chapter 37 is dedicated to it.
- Everything is **branch-per-version**: the Odoo source, the docs, OCA repositories,
  and your own projects all have `17.0`, `18.0`, `19.0` branches. A module written for
  one version does not run unmodified on the next.

This tutorial targets **Odoo 19.0 Community**. Where 18 behaves differently, you'll
see an "On Odoo 18" callout — useful when your client project runs a version behind.

### Where Odoo runs

Three hosting models, and the choice constrains what you can customize:

| | What it is | Custom modules? |
|---|---|---|
| **Odoo Online** | SaaS at odoo.com, Enterprise only | No — configuration and Studio only |
| **odoo.sh** | Odoo's PaaS: git-driven, per-branch staging builds | Yes |
| **On-premise / self-hosted** | You run the server (bare metal, VMs, Docker, k8s) | Yes — full control |

Integrators mostly live in the last two. This tutorial self-hosts with Docker, which
is also the closest to how professional teams run client projects.

## Gotchas

- **Odoo versions are not semver.** "19.0" is a product generation, not a
  major.minor you can partially upgrade across. Data and code migrate between
  generations via explicit migration work.
- **Community is not a demo.** It's the complete framework — some of the largest Odoo
  installations run pure Community plus OCA modules.
- **"Odoo" the company vs "Odoo" the software.** The company (Odoo SA, Belgium) builds
  the software and sells Enterprise/hosting. The next chapter maps the whole ecosystem
  around it.
