## Why this matters

Every confusing moment you'll have in the next 37 chapters — "where does this view come
from?", "why did my change not appear?", "what runs on the server vs in the browser?" —
resolves instantly if your mental model of the architecture is right. Fifteen minutes
here saves hours later. Nothing in this chapter needs memorizing; it needs *believing*,
and the rest of the tutorial keeps confirming it.

## Concepts

### Three tiers

<Mermaid chart={`
flowchart LR
    subgraph B["Browser"]
        OWL["Web client — OWL components\n(JavaScript SPA)"]
    end
    subgraph S["Odoo server (Python)"]
        HTTP["HTTP layer\n(controllers, RPC)"]
        ORM["ORM + business logic\n(models in modules)"]
    end
    DB[("PostgreSQL\nthe only supported database")]
    OWL <-->|"JSON-RPC over HTTP"| HTTP
    HTTP --> ORM
    ORM <-->|"SQL"| DB
`} />

- **Presentation** — the web client is a single-page JavaScript application built with
  **OWL** (Odoo Web Library), Odoo's own component framework — conceptually similar to
  React, but its own thing. It renders list views, forms, kanbans... from
  *descriptions* it receives from the server.
- **Logic** — a Python server. Business objects are Python classes; the **ORM** maps
  them to PostgreSQL tables and mediates every read and write. Alongside the RPC
  endpoints the web client uses, **controllers** serve classic HTTP routes (the
  website, the customer portal, PDF downloads).
- **Data** — PostgreSQL, and only PostgreSQL. Odoo leans on it hard (constraints,
  sequences, savepoints), which is why chapter 9 has you inspecting what the ORM
  creates with `psql`.

### Everything is a module

An Odoo server on its own does almost nothing. All functionality — including the core
`base` module that defines users, menus and models like `res.partner` — arrives as
**modules**: directories containing a manifest, Python code, XML data, and static
assets. The server discovers them via the **addons path**, and a database records which
modules are *installed in it*.

Two consequences worth internalizing now:

1. **Installation is per-database.** The same server can host one database where CRM
   is installed and another where it isn't. "Is the module installed?" is a question
   about a database, not a server.
2. **Modules extend each other.** A module can add fields to another module's models,
   inject elements into its views, and override its logic — without touching its
   source. That inheritance machinery (Part 4) is the single most important thing that
   makes Odoo customizable.

### The UI is data

Here is the idea newcomers take longest to absorb: **views are not template files.**
A form view, a list view, a menu item, an action — each is a *record in the database*
(in models like `ir.ui.view`), typically loaded there from a module's XML files at
install time. The web client asks the server "give me the form view for `res.partner`",
receives an XML description, and renders it.

That's why customizing Odoo rarely means forking anything: you ship a module whose XML
*adds records* — a new view, a menu — or *inherits existing ones* with targeted edits.

### One request, end to end

What actually happens when you open a customer and click **Save**:

<Mermaid chart={`
sequenceDiagram
    participant U as Browser (OWL)
    participant H as HTTP layer
    participant O as ORM
    participant P as PostgreSQL
    U->>H: JSON-RPC: res.partner.web_save(id, {changed fields})
    H->>H: authenticate session, open DB cursor
    H->>O: call model method
    O->>O: access rights check, business logic, computed fields
    O->>P: UPDATE res_partner SET ...
    P-->>O: ok
    O-->>H: record data
    H->>P: COMMIT
    H-->>U: JSON result
    U->>U: re-render form with fresh data
`} />

The details will fill in over the coming parts, but notice the shape: the browser never
talks SQL, the ORM never trusts the browser, and one HTTP request is one database
transaction — it commits when the request succeeds and rolls back if anything raises.
That last property will save you many times: a crashed request leaves no half-written
data behind.

## Hands-on

Reading source counts as hands-on in this chapter:

1. Open [`odoo/odoo` (19.0)](https://github.com/odoo/odoo/tree/19.0) and look at the
   top level: `odoo/` is the framework (ORM, HTTP, tooling), `addons/` is ~350
   modules — everything else. The split mirrors this chapter's story.
2. Inside [`addons/crm`](https://github.com/odoo/odoo/tree/19.0/addons/crm), find:
   `__manifest__.py`, a `models/` directory (Python — tier 2) and a `views/` directory
   (XML descriptions — the "UI is data" idea).
3. Open one file from `views/` and skim it: you're looking at records about to be
   loaded into `ir.ui.view`, not templates.

## Gotchas

- **"Odoo is a Python app" is only a third true.** When something misbehaves, first ask
  *which tier* — a JS error in the browser console, a Python traceback in the server
  log, and a constraint error from PostgreSQL are three different investigations.
- **The database remembers.** Because views and menus are records, uninstalling your
  code doesn't always remove what it loaded, and editing XML doesn't change a database
  that already loaded the old version — that's why the module *upgrade* step exists
  (chapter 8).
- **QWeb appears in two places.** The same XML templating language renders OWL
  components in the browser and reports/website pages on the server. Same syntax,
  different runtimes — keep the tiers straight and it won't confuse you.
