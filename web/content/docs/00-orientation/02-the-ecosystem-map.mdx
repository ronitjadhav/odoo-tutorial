**Goal:** know who the players are and which source to trust for what, so you never
google blindly again. · **Time:** ~1 h · **Checkpoint:** none, Part 0 has no code.

## Why this matters

Odoo development has an unusual amount of *ecosystem*: an editor company, a worldwide
partner network, a nonprofit with hundreds of community repositories, and several
generations of documentation. Knowing the map saves you hours weekly. And if you work
at an integrator, the OCA part isn't optional culture, it's your daily toolbox.

## Concepts

### The players

<Mermaid chart={`

- **Odoo SA** is the editor. It writes the core, ships the yearly release, and runs
  odoo.com, Odoo Online, odoo.sh, and the Enterprise subscription business.
- **Partners / integrators** are companies that implement Odoo for clients:
  requirements, configuration, custom modules, migrations, support. This is where most
  Odoo developers work (Camptocamp, where this tutorial's author works, is one of the
  larger ones).
- **The OCA (Odoo Community Association)** is a Swiss nonprofit that maintains hundreds
  of open-source module repositories at
  [github.com/OCA](https://github.com/OCA), organized by domain (`server-tools`,
  `web`, `partner-contact`, `account-financial-tools`, ...). Each repo is governed by a
  **PSC** (Project Steering Committee) and enforces serious quality gates: CI,
  peer review, conventions. Part 6 of this tutorial is entirely about working the
  OCA way.
- **You, eventually.** The OCA runs on contributions from integrator developers.
  Reusing an OCA module instead of writing your own is standard practice; contributing
  fixes back is expected etiquette.

### Where the knowledge lives

| Source | Use it for |
|---|---|
| [Official docs](https://www.odoo.com/documentation/19.0/), Developer section | Tutorials and reference for the framework. URLs are version-pinned, so always match your version. |
| [odoo/odoo source](https://github.com/odoo/odoo) | The ultimate reference. Reading core modules is a skill this tutorial trains deliberately. |
| [OCA repositories](https://github.com/OCA) | Production-grade examples of *community* best practice, often better study material than core. |
| [Odoo Apps store](https://apps.odoo.com) | Third-party modules, free and paid. Quality varies wildly, so check maintenance and reviews. |
| [Official YouTube](https://www.youtube.com/@Odoo) & [OCA Days talks](https://www.youtube.com/@OCAOdooCommunityAssociation) | Functional walkthroughs (official) and deep technical/community talks (OCA). |
| [Odoo forum](https://www.odoo.com/forum/help-1) & r/Odoo | Q&A. Check the answer's Odoo version before trusting it. |

<Callout type="warn" title="The version trap">
  The single most common way to lose an afternoon: reading docs, a forum answer, or a
  blog post written for a different Odoo version. Always check the version first.
  The framework changes every year, and pre-14 answers describe practically a
  different product.
</Callout>

### How code flows: an OCA repo up close

OCA repositories mirror Odoo's branch-per-version model. A repo like
`OCA/server-tools` has branches `16.0`, `17.0`, `18.0`, `19.0`: one line of
development per Odoo version, and modules get **migrated** branch to branch by
volunteers. Pull requests target a version branch (never a `master`), must pass CI and
pre-commit linting, get reviewed by peers, and are test-driveable on **Runboat**,
which spins up a throwaway live instance per PR.

You don't need to memorize this yet; you'll live it in Part 6. For now, just recognize
the shape.

## Hands-on

1. Open [github.com/OCA](https://github.com/OCA) and find the repository whose domain
   matches fleet/vehicle management. Note how repos are named by *domain*, not by
   module.
2. In [`OCA/server-tools`](https://github.com/OCA/server-tools), switch to the `19.0`
   branch. Compare the module list with the `16.0` branch. The difference is
   migration work not yet done (or no longer needed).
3. Open any merged PR in that repo. Read the review comments and notice: commit
   message format (`[FIX]`, `[IMP]`, `[MIG]`...), CI checks, reviewer sign-offs.
4. Browse [apps.odoo.com](https://apps.odoo.com) for "fleet" and compare what you find
   with the OCA equivalents: price, license, maintenance status.

## Verify

You should be able to say where you'd look first for: (a) the reference on a framework
feature, (b) a ready-made module for a common business need, (c) an example of
professionally-written module code, (d) an answer to "why does X behave like this".
(Answers, roughly: docs, then OCA/apps store, then OCA source, then forum/source.)

## Gotchas

- **The docs have generations too.** URLs containing `/documentation/19.0/` are
  version-pinned; if you land on a different version via search, switch it in the
  page's version selector.
- **Odoo Apps store ≠ OCA.** The store hosts anyone's modules, including abandoned
  ones. OCA modules pass review and CI; store modules might not have either.
- **PSC, ocabot, Runboat, Weblate**: OCA jargon you'll meet in Part 6. They're in the
  [glossary](/docs/glossary) already if you're curious now.
