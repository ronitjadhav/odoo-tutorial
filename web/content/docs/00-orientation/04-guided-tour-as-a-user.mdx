**Goal:** spend an hour inside Odoo as a user, then switch on developer mode and peek
behind the curtain. · **Time:** ~1.5 h · **Checkpoint:** none — but your Odoo runs!

## Why this matters

You cannot develop a product you've never used. Every model you'll build in Part 2
(customers, orders, stages, activities) is a concept Odoo's own apps already express —
and once you've clicked through them, the framework's vocabulary stops being abstract.
Developers who skip this stage write modules that fight the platform; developers who
know the product write modules that feel native.

## Hands-on

### 1. Start Odoo (on trust, for now)

The next chapter explains this setup line by line; today, just run it. You need
[Docker](https://docs.docker.com/get-docker/) installed.

```console
$ git clone https://github.com/ronitjadhav/odoo-tutorial.git
$ cd odoo-tutorial/code
$ docker compose up
```

Wait for the log to settle, then open [http://localhost:8069](http://localhost:8069).
You'll land on the **database manager**.

### 2. Create a database — with demo data

Fill the form: master password `admin`, database name `tour`, pick your email/password
for the admin user, and — **important** — check **Demo data**. Demo data fills the
database with a fictional company (customers, products, open orders), which is exactly
what makes clicking around instructive.

<Callout type="warn" title="Demo data is a birth decision">
  A database gets demo data at creation or never — there's no switch to flip later.
  If you forgot the checkbox, drop the database in the manager and create it again.
  (From the command line, Odoo 19 creates databases **without** demo unless you pass
  `--with-demo` — a default that changed over the years, so old tutorials disagree.)
</Callout>

### 3. Install two apps

You land in the **Apps** menu. Install **CRM**, then **Sales**. Watch what happens to
the top-left menu after each install — entire sections of the product appear. That's
the modularity from chapter 3, live: each install loaded a module's models, views and
menus into *this* database.

### 4. Be a salesperson for 20 minutes

No checklist pressure — the goal is exposure, not mastery:

- **CRM → My Pipeline.** This is a *kanban view* grouped by stage. Drag an opportunity
  to another column. Open one: the trail of messages and events at the bottom
  is the **chatter** — you'll add it to your own models in chapter 23.
- Create a new opportunity, schedule an activity on it, mark it won.
- **Sales → Orders → Quotations.** Open one, note the workflow buttons (Send, Confirm)
  and the status bar — states and state-driven buttons are chapter 17 material.
- Create a quotation for any demo customer, add products, confirm it, and notice the
  smart buttons that appear linking to related documents.
- Explore **Settings → Users & Companies → Users**: access rights grouped by app —
  that's the security model of chapter 10 seen from the admin side.

### 5. Switch on developer mode

Go to **Settings → General Settings**, scroll to the bottom, and under *Developer
Tools* click **Activate the developer mode**. (The shortcut you'll actually use daily:
append `?debug=1` to any backend URL.)

Three things just changed:

- A 🐞 icon appeared in the top bar — the developer tools menu.
- **Settings** grew a **Technical** section — the database's guts: views, actions,
  models, scheduled jobs, all browsable as ordinary records.
- Form fields now reveal their internals: open any customer, hover a field, and the
  tooltip shows its technical name (`phone`, `user_id`), its type, and which module
  defined it.

### 6. Peek behind one field

With developer mode on, open **Settings → Technical → Database Structure → Models**
and find `crm.lead`. Every field of the opportunity form you used ten minutes ago is a
row here. Now the circle closes: *the UI you clicked is described by data you can
browse*. In five chapters, rows in this list will be yours.

## Verify

Your database should show it all happened. From `code/`:

```console
$ docker compose exec db psql -U odoo -d tour -c "select count(*) from crm_lead;"
 count
-------
    45
(1 row)
```

Dozens of demo opportunities plus the ones you created (the demo set gave us 44 on a
fresh install — your exact number will differ). If the count is near zero, the demo
checkbox was missed — recreate the database.

## Gotchas

- **Developer mode is per session, not per server.** It follows your login session;
  activating it doesn't affect other users, and `?debug=0` turns it off.
- **`debug=assets` exists too.** It serves unminified JS/CSS — you'll want it in
  Part 5 when debugging OWL; it makes everything slower, so don't leave it on.
- **Demo data is not sample data for production.** It exists for training, tests, and
  Runboat-style throwaway instances. Client projects always start without it.
- **Don't study Enterprise-only apps yet.** If you see apps in the list that ask for a
  subscription code (Studio, some Accounting features), skip them — everything in this
  tutorial runs on Community.

## Exercises

1. Install **Inventory** and trace what your confirmed sales order created there
   (hint: the smart button on the order).
2. With developer mode on, find which module defined the `expected_revenue` field on
   `crm.lead` — using only the UI.
3. In the Technical menu, find the *form view* of `crm.lead` and skim its XML in the
   Architecture tab. You already recognize more of it than you'd expect after
   chapter 3.
