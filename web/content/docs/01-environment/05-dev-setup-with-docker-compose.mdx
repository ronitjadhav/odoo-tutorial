**Goal:** run Odoo 19 + PostgreSQL with Docker Compose and understand every line that
makes it work. · **Time:** ~1.5 h · **Checkpoint:** none, but odoolings checks start
here.

## Why this matters

In chapter 4 you started this environment "on trust". Today the trust runs out: this
is the workbench you'll use for the next 35 chapters, and when it misbehaves (a port
collision, a dead database, a module that won't load), you fix it fastest by knowing
exactly what each piece does.

Docker Compose is also not a tutorial convenience. It mirrors how integrators actually
run client projects: same containers, same config file idea, same separation between
the Odoo image and *your* addons mounted into it. What changes in production is
scale and secrets, not shape.

## Concepts

### Two containers, one file

Everything lives in [`code/docker-compose.yml`](https://github.com/ronitjadhav/odoo-tutorial/blob/main/code/docker-compose.yml):

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_USER: odoo
      POSTGRES_PASSWORD: odoo
      POSTGRES_DB: postgres
    volumes:
      - db-data:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U odoo"]
      interval: 5s
      retries: 10

  odoo:
    image: odoo:19
    depends_on:
      db:
        condition: service_healthy
    ports:
      - "8069:8069"
    volumes:
      - odoo-data:/var/lib/odoo
      - ./odoo.conf:/etc/odoo/odoo.conf:ro
      - ./addons:/mnt/extra-addons

volumes:
  db-data:
  odoo-data:
```

Read it top to bottom:

- **`db`** runs PostgreSQL 16. The `POSTGRES_USER`/`POSTGRES_PASSWORD` pair creates a
  database role called `odoo`, which is what the Odoo server logs in with. The
  `healthcheck` matters more than it looks: Odoo crashes at startup if Postgres isn't
  accepting connections yet, so...
- **`depends_on` + `condition: service_healthy`** makes Compose hold the Odoo
  container back until `pg_isready` succeeds. Without it you get a race and a
  confusing traceback on cold starts.
- **`ports: "8069:8069"`** publishes Odoo's HTTP port to your machine. 8069 is Odoo's
  default and every Odoo developer has it memorized.
- **Named volumes** (`db-data`, `odoo-data`) are where the *state* lives: the Postgres
  data directory and Odoo's **filestore** (attachments and uploaded files live on
  disk, not in the database). Containers are disposable; these volumes are not.
- **Bind mounts** are the developer magic: `./odoo.conf` injects our config file, and
  `./addons` (an empty directory today) is where LibreFleet will live from chapter 8.
  Code on your machine, running inside the container.

### The config file

Odoo reads `/etc/odoo/odoo.conf`, which is our
[`code/odoo.conf`](https://github.com/ronitjadhav/odoo-tutorial/blob/main/code/odoo.conf):

```ini
[options]
addons_path = /mnt/extra-addons
db_host = db
db_user = odoo
db_password = odoo
; dev-only credentials, hardcoded on purpose. Never reuse this file in staging/prod.
admin_passwd = admin
dev = all
```

- **`addons_path`** tells the server where to look for extra modules, and it points at
  the bind mount above. The official image already knows about its own built-in
  addons; this adds ours.
- **`db_host = db`** is Compose networking in action: services reach each other by
  service name. No IP addresses, no localhost.
- **`admin_passwd`** is the **master password**, the one the database manager screen
  asks for when you create, duplicate or drop databases. It is *not* a user's login
  password.
- **`dev = all`** turns on developer conveniences, most importantly auto-reload:
  the server picks up Python changes without a manual restart. Chapter 6 tours what
  else it does.

  The [configuration file reference](https://www.odoo.com/documentation/19.0/developer/reference/cli.html)

The split to internalize: **code is yours and git-tracked on the host; state (databases
and filestore) belongs to Docker volumes.** You can destroy and rebuild containers
freely. Destroying volumes is destroying your databases.

## Hands-on

### 1. Get it running

You need [Docker with the Compose plugin](https://docs.docker.com/get-docker/)
(Docker Desktop on macOS/Windows, `docker-ce` + `docker-compose-plugin` on Linux).
If you did chapter 4, you've already cloned the repo.

```console
$ git clone https://github.com/ronitjadhav/odoo-tutorial.git
$ cd odoo-tutorial/code
$ docker compose up
```

The first run downloads both images (a few minutes). You're waiting for one line:

```text
odoo-1  | ... INFO ? odoo: Odoo version 19.0-20260630
odoo-1  | ... INFO ? odoo.service.server: HTTP service (werkzeug) running on ...:8069
```

`Ctrl-C` stops it. Prefer running it detached and reading logs on demand:

```console
$ docker compose up -d
$ docker compose logs -f odoo    # Ctrl-C stops following, not the server
```

### 2. Create your working database

Open [http://localhost:8069](http://localhost:8069). In the database manager, create
the database you'll build LibreFleet in. These exact values are what `odoolings`
expects by default, so use them unless you enjoy typing flags:

| Field | Value |
|---|---|
| Master password | `admin` |
| Database name | `tutorial` |
| Email (login) | `admin` |
| Password | `admin` |
| Demo data | **unchecked** |

No demo data this time. The `tour` database from chapter 4 was for exploring; this
one is your workshop, and client projects start empty too. Log in and you'll see a
nearly bare Odoo. Perfect.

### 3. Break it and fix it (on purpose)

Your first break-it lab. Before touching anything, read these two commands closely:

```console
$ docker compose down        # stops and removes CONTAINERS. State survives.
$ docker compose down -v     # also removes VOLUMES. Databases gone. Filestore gone.
```

Run the first one, then `docker compose up -d` again, refresh the browser, and
confirm the `tutorial` database survived. Now you know a container restart is always
safe. Leave `-v` for the day you *want* a factory reset. (If you just ran `-v`
anyway: congratulations, you've proven the point. Recreate the database.)

### 4. Look inside the containers

Two commands you'll use constantly. A shell inside the Odoo container:

```console
$ docker compose exec odoo bash
odoo@42aabd6856a0:/$ ls /mnt/extra-addons     # your (empty, for now) addons dir
```

And `psql` inside the database container:

```console
$ docker compose exec db psql -U odoo -d tutorial
tutorial=# select count(*) from ir_module_module where state='installed';
 count
-------
    14
(1 row)
```

Fourteen modules installed and you haven't done anything: that's `base` and its
minimal court. From chapter 9 on, `psql` is how we check what the ORM is really doing.

## Verify

Time to meet `odoolings`, the checker you'll run at the end of every hands-on from now
on. It lives in `code/` and needs nothing but Python 3:

```console
$ python3 odoolings.py check ch05
✔ Odoo server is reachable
✔ server version is 19.x
✔ can log in as admin

ch05 complete! On to the next chapter 🔧
```

All three green means your workbench is exactly what the rest of the tutorial assumes.
If something is red, the hint under it tells you where to look.

## Gotchas

- **`down -v` is the only destructive command in your daily set.** `stop`, `down`,
  `restart`, rebooting your machine: all safe, state lives in volumes. Treat `-v`
  like `rm -rf`.
- **Port 8069 already in use?** Another Odoo (or an old container) is running. Either
  stop it, or change the mapping to `"8169:8069"` and browse to `:8169`. The part
  left of the colon is your machine; the right is the container.
- **Config changes need a restart.** `odoo.conf` is read at startup, so run
  `docker compose restart odoo` after editing it. (Python code changes don't need
  this, thanks to `dev = all`.)
- **Linux file permissions.** The container runs as its own `odoo` user. If module
  files you create later show up as unreadable, check ownership on `code/addons`;
  keeping it world-readable is enough for dev.
- **`docker-compose` vs `docker compose`.** The hyphenated v1 binary is dead. If your
  system only has `docker-compose`, your Docker is old. Update it rather than
  fight it.

## Exercises

1. ⭐ Duplicate your `tutorial` database from the database manager (it asks for the
   master password) and then drop the copy. Databases are cheap; get comfortable
   making and destroying them.
2. ⭐⭐ Change the published port to 8169, apply the change with the right compose
   command, verify in the browser, then change it back. Bonus: why does `restart`
   alone not work for this one? (Hint: ports are container *creation* settings.)
3. ⭐⭐⭐ Run a second, fully separate Odoo playground in another directory with its
   own compose file, own volumes, and port 8169. When it works, throw it away with
   the destructive command, on purpose this time.
