## Why this matters

You already know git. What's new is how the Odoo world uses it: one convention
(a branch per Odoo version) shapes every repository you'll touch, and integrator
projects are mostly *assemblies* of other people's modules, which makes dependency
pinning a first-class concern. Get this mental model now, and OCA repos, client
projects, and your own tutorial work all look organized instead of arbitrary.

## Concepts

### Branch-per-version, everywhere

An Odoo module is written *for* one Odoo version (chapter 1). Git encodes that as a
branch named after the version, and the convention is universal:

- `odoo/odoo` has branches `17.0`, `18.0`, `19.0`, plus `master` where the next
  release is developed.
- Every OCA repo has the same branches. The same module exists on several of them,
  and moving it forward is an explicit **migration** commit (`[MIG]`).
- Client projects follow suit: when the client upgrades from 18 to 19, the project
  repo grows a `19.0` branch and the porting work happens there.

Two rules fall out of this: **never target `master`** with a fix (that's next year's
Odoo), and **a branch name is a compatibility statement**, so code on `19.0` must run
on Odoo 19, full stop.

Here's the model measured for real. The `OCA/server-tools` repo, same repo, two
branches:

```console
$ git clone --depth 1 -b 19.0 https://github.com/OCA/server-tools.git st-19
$ git clone --depth 1 -b 16.0 https://github.com/OCA/server-tools.git st-16
$ ls st-19 | wc -l
32
$ ls st-16 | wc -l
73
```

Same repository, 73 modules on the mature 16.0 branch, 32 on the young 19.0 branch.
The difference is the migration queue: modules nobody has needed (or volunteered) on
19 yet. When a client project needs one of the missing ones, an integrator developer
does the `[MIG]`, and that developer will regularly be you.

### How commit messages look

Odoo and OCA share a commit convention you'll adopt in Part 6, but should start
reading now. Real messages from `server-tools` 19.0:

```text
5dcfe30 [FIX] database_cleanup: restrict cleanup tools to settings users
63d1654 [MIG] base_search_fuzzy: Migration to 19.0
f0a3744 [IMP] module_auto_update: confirm message before updating
```

The shape is `[TAG] module_name: what and why`. Common tags: `[FIX]` bugfix, `[IMP]`
improvement, `[ADD]` new module/feature, `[MIG]` migration to a version, `[REF]`
refactor. A `git log --oneline` on any OCA module reads like a changelog, which is
the point.

### What a client project repo looks like

An integrator's project repo rarely contains much code. It's an assembly manifest:

```text
client-project/            # branch: 19.0
├── addons/                # the project's own custom modules (a handful)
├── external-src/          # OCA & third-party repos, pinned (see below)
├── odoo/                  # Odoo itself: a pinned checkout, or nothing if the
│                          #   Docker image provides it (our setup)
├── docker-compose.yml / Dockerfile
└── requirements.txt / repos.yml / .gitmodules   # the pinning mechanism
```

The custom `addons/` part stays small on healthy projects because the first question
is always "does an OCA module already do this?" (chapter 2). The bulk of the
assembly is *other people's modules at exact versions*.

### Pinning: never deploy a moving target

`19.0` branches move daily. If your production image checks out "whatever
`OCA/server-tools` 19.0 is today", every deploy is a surprise. So professional
projects pin, using one of three mechanisms:

| Mechanism | How it pins | Feel |
|---|---|---|
| **git submodules** | a SHA per external repo, recorded in `.gitmodules` | native git, clunky UX |
| **git-aggregator** | `repos.yml` listing repos + SHAs/PRs to merge | OCA tooling, powerful (can stack unmerged PRs) |
| **pip** | OCA publishes every module as `odoo-addon-<name>`; pin in `requirements.txt` | plain Python packaging |

<Callout title="In the field">
  [git-aggregator](https://github.com/acsone/git-aggregator) is widespread among OCA
  contributors and integrators (including Camptocamp): one YAML file describes every
  external repo, the commit to pin, and even pending PRs to merge into your build,
  which is how you ship a fix the day you write it instead of after the OCA review
  cycle. Ask your team which mechanism your projects use; the concept transfers.
</Callout>

Whatever the tool, the invariant is the same: **a production build is reproducible
from a commit of the project repo alone.** If `git checkout` + build can produce two
different deployments on two days, the project is broken.

### Where your tutorial work fits

You cloned `ronitjadhav/odoo-tutorial`, and from chapter 8 your own LibreFleet code
grows in `code/addons/`. Committing your work to someone else's clone gets awkward,
so make the workspace yours today, before there's code:

- **Fork** the tutorial repo on GitHub and point your clone at it (hands-on below).
  Your work lives on your fork's `main`; pulling upstream updates stays one command.
- From chapter 8 on, **commit after every chapter** using the OCA message style
  (`[ADD] librefleet: vehicle model`). By Part 6 the convention will be muscle
  memory, and your git history becomes your own set of checkpoints.

## Hands-on

### 1. Make the workspace yours

On GitHub, fork `ronitjadhav/odoo-tutorial`. Then, in your existing clone:

```console
$ git remote rename origin upstream
$ git remote add origin git@github.com:<you>/odoo-tutorial.git
$ git push -u origin main
$ git remote -v      # origin = your fork, upstream = the tutorial
```

From now on: `git pull upstream main` brings tutorial updates, `git push` saves your
work. (If the pull ever conflicts with your LibreFleet files, yours win; checkpoints
exist precisely so you can diff against the reference.)

### 2. Measure the migration queue yourself

Run the two shallow clones from the Concepts section and count modules per branch.
Then pick one module that exists on `16.0` but not on `19.0`, and check its
`__manifest__.py` on the old branch: would *you* need it on a project?

### 3. Read a module's history like a changelog

```console
$ cd st-19
$ git log --oneline -15 -- database_cleanup
```

Identify each tag you see and notice how much you understand about the module without
opening a single source file. That's what disciplined commit messages buy.

### 4. Practice the message style on yourself

Commit something harmless in your fork (fix a typo in your notes, add a `.gitignore`
entry) with a properly shaped message, e.g.
`[IMP] docs: add scratch notes directory to gitignore`. Shape, module, what, why.

## Verify

`git remote -v` shows `origin` (your fork) and `upstream` (the tutorial). `git push`
works. You can say out loud why `OCA/server-tools` has fewer modules on `19.0` than
on `16.0`, and what `[MIG]` means. No odoolings checks this chapter; git lives
outside the Odoo server.

## Gotchas

- **`master` is a trap.** On `odoo/odoo` and OCA repos it's the unreleased next
  version. Code examples found on `master` may not work on 19, and PRs against it
  will be closed with a polite "wrong branch".
- **Branch names are floats, sort of.** `19.0` not `19`, `16.0` not `v16`. Tooling
  (and humans) rely on the exact `XY.0` form.
- **Submodule/aggregator checkouts are detached HEADs.** That's normal: pins point at
  SHAs, not branches. Don't "fix" it by checking out the branch inside a pinned repo.
- **One module, one concern, one history.** Resist the "misc" module that accretes
  everything; OCA repos are readable because each module's log tells one story.
  Chapter 8 starts LibreFleet with that discipline.
